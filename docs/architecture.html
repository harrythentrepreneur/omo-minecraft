Five layers

From the player to the outside world

1

You — in Minecraft

An ordinary vanilla world. You speak a goal, talk to any agent, steer with /revise, and veto any outward action with one tap. Push-to-talk (V) routes voice straight to the agents.

the spatial interface
2

The World — Java 21

The Paper plugin + Omo client-mod are the only code that touches Minecraft: live block-by-block builds, villager spawning & pathing, fullscreen CinemaScreen dashboards, and the tap-to-approve UI.

plugin/ · client-mod/
3

The Orchestrator — Node / TS

One event loop: the WS bridge, the brain-selecting AgentManager, the AdkAgent SSE bridge, the WorldStore org-graph, the live dashboard server, the Gemini world-architect, and the omo-tools MCP server that hosts the World API.

runtime/src/
4

The Organisation — ADK · Python

A real multi-agent system: a Gemini Chief of Staff that delegates to Growth & Comms specialists and hires new ones on demand. Served by adk api_server; tools arrive via McpToolset.

omo-agent/
5

The Outside World — over MCP

Every tool reaches the agents the same way — over MCP. Real Meta Ads data is live today; Gemini is the brain of every agent. Google Workspace & Stripe MCP are the roadmap.

The novelty

The self-extension loop

You never pre-wire a function. You state intent — the org declares the function it needs and the world materialises it, all by calling MCP tools:

1. world_describe→2. world_add_function→3. world_build→4. world_staff→5. world_assign

★ Plus world_consult — staffed functions ask each other questions instead of guessing, so the org behaves like a team, not a star of isolated workers.

Mandatory stack

Three technologies, each load-bearing

Gemini the brain of every agent

gemini-flash-latest runs the Chief of Staff, every specialist, and the architect that designs each building.

ADK the organisation

An LlmAgent coordinator with sub_agents that delegate via transfer_to_agent, streamed token-by-token over /run_sse.

MCP every tool — and self-extension

A Streamable-HTTP server exposes real tools and the World API, so the org grows itself by calling an MCP tool — not by shipping code.
